01 — product decisions
Decision 01 —

Microservices over monolith from Day 1.

Client had a clear roadmap to integrate a large suite of vendor services post-launch — inventory management, analytics, subscription billing. A monolith would have required a painful refactor at scale. We chose a Gateway + App Server + Dashboard Server pattern to handle consumer and vendor request traffic independently.

// PM call
forward-looking architecture, not over-engineering. justified by client's explicit roadmap for vendor suite expansion.
Decision 02 —

Checkout App as a route, not a new application.

The checkout flow for security personnel was simple — scan a QR code, verify the transaction, confirm. Building a standalone app would have added infra overhead, separate deployment, and maintenance cost with zero benefit. It became a protected route in the main dashboard app.

// PM call
scope discipline. don't build a new app when a route solves the problem. saved ~2 weeks of dev and eliminated a deployment surface.
Decision 03 —

Apollo GraphQL federation for multi-surface data.

Three surfaces with overlapping data needs — user orders, vendor inventory, security verification — but different access patterns. GraphQL's declarative fetching meant each surface asked for exactly what it needed. Apollo Client middleware normalised and cached data on the frontend, reducing redundant network calls.

// tech call
graphql schema as contract between frontend and backend. both teams work from one source of truth.
Decision 04 —

React Native for Android-first, iOS-ready.

Client needed an Android app at launch. React Native gave us a single codebase that ships to both platforms — meaning iOS becomes a config change, not a rebuild. Shared component logic with the React dashboard also reduced cognitive overhead across the team.

// PM call
cross-platform from the start. no re-platforming cost when iOS enters the roadmap.
Decision 05 —

Firebase Auth + MongoDB split by concern.

Firebase handles authentication — identity, tokens, multi-factor, session management. MongoDB handles application data — orders, vendors, inventory, payments. This is a deliberate separation: auth concerns never pollute the data model, and swapping the auth provider later doesn't touch the database schema.

// tech call
single responsibility at the infra level. auth is a solved problem — use a service. data is your moat — own it.
Decision 06 —

Free tier ad model with hard expiry at v1.

The ad platform launched with a free tier — 3 ads per type, 3-day expiry. This was a conscious PM decision: validate advertiser behaviour and willingness before building subscription billing. Subscription plans were explicitly moved to v2, listed in the release notes as future features.

// PM call
ship the simplest monetisation model that generates signal. don't build billing before you know people will pay.
02 — system architecture
Microservice Architecture click to expand
// gateway · app server · dashboard server · mongodb
⬡

Gateway Server as single entry point

All client requests route through the Gateway. It decides whether to hit the App Server (consumer data) or Dashboard Server (vendor/admin data). This is where auth tokens are validated before any data is fetched.

// PM implication: vendor and consumer scaling is now independent
◈

Apollo Client Middleware on the frontend

The middleware layer normalises and caches GraphQL responses on the client. UI updates happen from the local cache — the server is only hit when the cache is stale. This makes all three surfaces feel fast even on poor connections.

// PM implication: performance without infra cost
◎

Firebase Auth as a side channel

Auth is completely separate from the data graph. Firebase handles signin and token issuance. The token is validated at the Gateway. Application data never touches Firebase — MongoDB owns everything else.

// PM implication: auth provider is swappable without data migration
Application Architecture — Single Server View v1.0 shipped
// why this architecture was the right call

At v1 scale, the single-server view and the microservice view look similar. The difference shows up at v2 when vendor suite services are added. Because the Dashboard Server is already a separate service behind the Gateway, new vendor services slot in without touching the consumer-facing App Server. That's the PM decision embedded in the architecture — build for the roadmap, not just the launch.

03 — data model
ER Diagram click to expand
Schema decision

Soft deletes on Store and Personnel

Store and Personnel models carry deletedBy and deletedDate fields. Hard deletes would break order history references. Soft deletes preserve data integrity across historical records while removing entities from active views.

// PM implication: audit trail is preserved for dispute resolution
Schema decision

Vendor → Store → Personnel hierarchy

A vendor can own multiple stores. Each store has its own personnel with role-based access. This maps to how retail chains actually operate — one business entity, multiple physical locations, different staff at each.

// PM implication: supports multi-location vendor onboarding in v2 without schema changes
Schema decision

Order History separate from Order

Active orders and historical orders are separated. Order is transactional — it changes state frequently. Order History is append-only — it records what happened. Separating them keeps the active order collection lean and fast.

// PM implication: shopping history feature loads fast regardless of lifetime order volume
Schema decision

ProjectSetting as a config collection

App name, logo, tagline, support email — all in a single document. Admin can update branding without a deployment. This was a conscious decision to separate config from code.

// PM implication: white-labelling this product for another client is a DB update, not a code change

04 — scope discipline
What we didn't build in v1 cut

Everything that wasn't ready to validate.

These features were explicitly scoped out of v1 and documented in release notes as future work — not forgotten, deliberately deferred.

  • Location-based store display
  • Returns and replacements flow
  • Coin and rewards system
  • Referral code integration
  • Full vendor inventory management
  • Comprehensive analytics + reporting
  • Advertisement scheduling
  • Subscription plan billing
  • Full microservices migration
What we shipped in v1 shipped

The smallest system that validated the core hypothesis.

Core user journey end-to-end. Vendor onboarding. Ad platform free tier. Security verification. Complete data model that supports v2 without schema changes.

  • Product scanning + cart management
  • Secure checkout via PhonePe
  • QR code receipt + security verification
  • Vendor + advertiser onboarding
  • Ad upload (Stories, Banners, Tiles)
  • Admin inventory management
  • Shopping history + receipts
  • Firebase Auth + role-based access
  • Full handover documentation
// the PM discipline behind the cut list

Every feature on the cut list has a reason it was deferred, not dropped. Location-based stores require GPS permission flows and store geofencing — adds complexity without validating the core checkout behaviour. Subscription billing requires knowing advertisers will pay — free tier generates that signal first. The cut list is a roadmap, not a backlog of failures.

05 — stack decisions
Mobile
React Native Single codebase for Android + iOS. Native performance via native UI components.
Web Frontend
React Dashboard and checkout app. Component reuse across both surfaces.
API Layer
Apollo GraphQL Declarative fetching. Schema as contract. Gateway federation for microservices.
Backend
Node.js + TypeScript Async event-driven — handles concurrent scan + checkout requests efficiently. TypeScript enforces type safety across services.
Database
MongoDB + Mongoose Flexible schema for evolving product data. Mongoose enforces validation and provides ODM query builder.
Auth + Storage
Firebase Auth Handles identity, tokens, MFA. Separated from application data intentionally.
Firebase Storage Ad media (stories, banners, tiles). Offloads binary storage from MongoDB.
Payments
PhonePe UPI-native payment integration. Generates QR receipt verified by security checkout app.
